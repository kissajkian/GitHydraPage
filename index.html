About Us

The motivation to create a new advanced organization model for Git, came with the aim of correct some existing gaps in the current available organization tools. Following are some of these gaps: Merge on the development branch at each release deployed on Production, Merge on release branches with production fixes that could be developed during client validation, Parallel development branches and so on.


In Greek Mythology, the Hydra was a monster that possessed many heads and had regeneration ability, making two heads grow if one of them was chopped off. GitHydra brings this concept, defining several development branches, working in parallel. With this, if any problem occur on some of the branches (analogy for the Hydra heads), there will be others to support the code.

To install Git Hydra on your Git Installation, you should put all content located unded "commons" and "core" folder, into your git-installation-folder\usr\bin folder.

We are working on a installation script and it should be available very soon.



All feature branches will start from one of the development branches. If your Hydra repository has only one development branch, it will start from there.

To start a feature, you should run the command "git hydra feature start feature_name [origin_dev_branch]". If your repository is already on a development branch, the second parameter is optional.

After you've finished your work, you can finish your feature by running "git hydra feature finish". This will merge your feature into your release branch. At this point, your development branch(es) is (are) clean with the production point.



All bug branches are created based on the release branch. These branches represent problems identified during client validation, before your code is deployed on Production Environment. Every bug branch are merged into release branch, after you've finished your work.

To start a bug branch, run the command "git hydra bug start", and to finish it, run "git hydra bug finish bug_name"



Similar to bug branches, these branches always start from your Production branch, and it represent errors found on Production environment.

There are two ways to work with these branches:

If your project does not have an exclusive fix environment to test the code from your fix development, you will find that is necessary to first merge your fix into your release Environment; to do this, run the command "git hydra fix fix_name release;

If your project have an exclusive fix environment to test the code from your fix development, after you test it, you can finish your branch, so it could be merged into all your main branches; to do this, run the command "git hydra fix finish fix_name".



This will be a main branch, that will live forever. All features, bugs and fixes will be merged into it. This branch is intended to represent the code that is being validated by someone, before its deployment on Production.

After all validation is finished and the deploy was made, you should run the command "git hydra release finish". This command will merge the code inside release branch, into your production branch and into all your development branches.
